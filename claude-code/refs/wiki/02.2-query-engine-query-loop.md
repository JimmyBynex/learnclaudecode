# 2.2 QueryEngine & Query Loop

## Core Components

| Component | Role |
|-----------|-----|
| `QueryEngine` | Stateful class holding message history, file cache, usage stats, task config |
| `submitMessage()` | Async generator yielding messages as LLM produces them, managing tool-calling loop |
| `ask()` | Primary SDK entry point instantiating QueryEngine for single-prompt execution |
| `query()` | Low-level Anthropic API interface: streaming, retries, response parsing |

## Query Lifecycle Flow
`submitMessage()` manages one "turn" (potentially multiple LLM calls if tools are invoked). Maintains message history, tracks token usage, handles context window management via compaction and snipping.

## System Prompt Assembly
Before each LLM request, engine constructs comprehensive system prompt combining:
- Base behavioral instructions
- Environment context (working directory, OS details)
- XML-formatted tool definitions
- Memory content from `MEMORY.md`
- Custom user-provided instructions

## Tool Execution Flow
When LLM returns tool calls:
1. Identify the tool and map to its definition
2. Check permissions via `canUseTool` hook
3. Execute tool logic
4. Wrap results in `tool_result` messages
5. Append to history and continue loop

## Context Management (Three Strategies)

| Strategy | Mechanism |
|----------|----------|
| Compact boundaries | Identified summarization points in history |
| Snipping | Truncate history while maintaining conversation state |
| Micro-compaction | Remove redundant metadata, shrink large outputs |
