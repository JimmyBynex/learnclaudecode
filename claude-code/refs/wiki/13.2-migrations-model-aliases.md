# 13.2 Migrations & Model Aliases

## Model Aliases & Configuration

Abstracts specific Anthropic model strings into logical aliases. CLI references `opus` or `sonnet` while engine resolves to correct API strings or cloud-specific identifiers.

### Key Components

| Component | Location | Function |
|-----------|---------|---------|
| Model Definitions | `src/utils/model/model.ts` | Supported models with capabilities (context window, tool support) and pricing |
| Model Aliases | `src/utils/model/antModels.ts` | Short names (e.g., `sonnet`) → full versioned strings |
| Provider Mapping | Translation logic | Amazon Bedrock and Google Vertex AI mapping |

### Core Functions
- `validateModel`: Verifies if string is known model or alias
- `getModelCapabilities`: Returns supported features for specific model
- `getProviderForModel`: Identifies provider (anthropic, bedrock, vertex)

## Migration System (`src/migrations/`)

Structured system transforming data between versions. Primary use: **Model Upgrading** — automatically updates user configs when new model versions release.

### Migration Workflow
- Bootstrap phase detection and application
- Model transition tracking (e.g., Fennec → Opus, Sonnet version upgrades)
- Runtime resolution via `QueryEngine`

### Implementation
- `ModelStrings` type in `src/bootstrap/state.ts`: maintains exact model config for sessions
- `STATE` object: runtime source of truth
- `mainLoopModelOverride`: triggers `QueryEngine` resolution when users run `/model opus`

## Deprecation System
- **Model Deprecation**: Marked models surface warnings during session initialization
- **Alias Redirection**: Old aliases transparently redirect to modern equivalents with deprecation notices

## Configuration Migrations
Beyond model changes:
- Schema restructuring (flat → nested structures)
- Permission rule format updates
- Telemetry identifier rotation
- Settings cache invalidation via `resetSettingsCache`
