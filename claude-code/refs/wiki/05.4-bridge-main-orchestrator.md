# 5.4 Bridge Main Orchestrator & Env-less Path

## Main Orchestrator Loop (`runBridgeLoop`)

Manages multiple concurrent sessions:
1. Worker registration (directory/machine name)
2. Continuous polling via `pollForWork`
3. Decode `workSecret`, launch child processes
4. Session tracking with heartbeat monitoring and timeouts
5. Capacity management — wait when `maxSessions` limit reached

Implements exponential backoff. Sleep/wake detection via polling delta analysis: "if delta exceeds 2x the connection cap, backoff is reset."

## Env-less Path (CCR v2)
Streamlined connection avoiding Environments API work-dispatch layer. For REPL-based Remote Control:
- Direct session creation via `POST /v1/code/sessions`
- JWT exchange at `/bridge` for `worker_jwt` and `worker_epoch`
- SSE (Server-Sent Events) for inbound; `CCRClient` for outbound
- `TokenRefreshScheduler` for proactive JWT renewal

## Session Creation — `createBridgeSession` (`src/bridge/createSession.ts`)
- Handles `POST /v1/sessions` for remote-control commands
- Injects `git_repository` info and `session_context` metadata
- Pre-populates history with `SDKMessage` objects wrapped as `SessionEvent`

## Session ID Compatibility
- Infrastructure IDs: `cse_` prefix for API calls
- Compatibility IDs: `session_` prefix for user-facing displays
- `toCompatSessionId` normalizes identifiers

## Session History API (`src/assistant/sessionHistory.ts`)
Paginated cursor-based retrieval:
- `HISTORY_PAGE_SIZE`: 100 events per request
- `fetchLatestEvents`: Uses `anchor_to_latest: true`
- `fetchOlderEvents`: Traverses backward using `before_id` cursor

## Key Types

| Type | Purpose |
|------|---------|
| `BridgeConfig` | Directory, machine name, spawn mode, bridge ID settings |
| `SpawnMode` | `single-session`, `worktree`, or `same-dir` strategies |
| `WorkResponse` | Poll API payload with work ID and secret |
| `HistoryPage` | Paginated event collection with metadata |
| `BackoffConfig` | Retry timing and shutdown grace periods |
