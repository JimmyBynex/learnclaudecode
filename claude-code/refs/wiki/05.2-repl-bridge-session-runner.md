# 5.2 REPL Bridge & Session Runner

## Overview
Communication layer connecting local Claude Code CLI to remote environments (Claude.ai web interfaces). Enables "Remote Control" mode — users manage local codebase through remote LLM interfaces while local agent executes tools.

## Key Components

| Component | Purpose | Location |
|-----------|---------|----------|
| `ReplBridge` | Primary coordinator: session lifecycle + message routing | replBridge.ts |
| `initReplBridge` | High-level init wrapper resolving environment state | initReplBridge.ts |
| `ReplBridgeTransport` | Abstraction for transport protocols (WebSocket V1 vs SSE/CCR V2) | replBridgeTransport.ts |
| `SessionRunner` | Spawns and manages CLI as child process for remote sessions | sessionRunner.ts |
| `FlushGate` | State machine preventing message interleaving during sync | flushGate.ts |

## Core Functions

**Message Processing:**
- Image normalization (correcting mobile/web client malformed blocks)
- Attachment resolution (fetching file_uuid attachments → `~/.claude/uploads/`)
- Ingress routing (parsing incoming data, deduplicating echoes, routing to handlers)

**FlushGate Lifecycle:**
1. `start()` — initiates flush, begins queueing
2. `enqueue()` — stores items if active
3. `end()` — returns queued items for draining

## Session Execution
`SessionRunner` uses `child_process.spawn` with `--sdk-url` and `--sdk-ingress-token` flags. Monitors stdout for NDJSON events. Tracks activity using `TOOL_VERBS` mapping (e.g., BashTool → "Running").

## Transport Protocols

| Version | Implementation | Method |
|---------|----------------|--------|
| V1 | HybridTransport | WebSockets for reads, HTTP POST for writes |
| V2 | SSETransport + CCRClient | Server-Sent Events for reads, batched writes to CCR endpoints |

## Session ID Compatibility
Translates between `cse_*` (infrastructure) and `session_*` (client-facing) ID formats via `sessionIdCompat.ts`, with feature flag control via `setCseShimGate`.
