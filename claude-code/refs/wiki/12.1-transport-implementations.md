# 12.1 Transport Implementations

## Overview
Transport layer manages bidirectional communication between Claude Code CLI and remote session infrastructure: protocol selection, event batching, retries, backpressure management.

## Transport Selection

| Transport | When Used |
|-----------|----------|
| `SSETransport` | `CLAUDE_CODE_USE_CCR_V2` enabled — SSE reads + HTTP POST writes |
| `HybridTransport` | `CLAUDE_CODE_POST_FOR_SESSION_INGRESS_V2` enabled — WebSocket reads + HTTP POST writes |
| `WebSocketTransport` | Default for `ws:`/`wss:` protocols |

## Core Transport Classes

### WebSocketTransport
- Keep-alive: periodic data frames (5-min default) + WebSocket pings (10-sec default)
- Automatic reconnection with exponential backoff using CircularBuffer (1000 message capacity)
- Sleep/wake detection via reconnection attempt gap analysis
- `X-Last-Request-Id` header for server-side stream resumption

### SSETransport (Cloud-native CCR v2)
- Incremental SSE frame parsing (handles multi-line data fields and keep-alive comments)
- Liveness monitoring: connection dead after 45 seconds without activity
- Stream resumption via `Last-Event-ID` header with sequence number tracking

### HybridTransport
- Extends WebSocketTransport with HTTP POST writes (prevents retry storms and Firestore collisions)
- Content delta buffering (100ms) to reduce POST volume during high-throughput streaming
- Single serialized uploader for all outgoing events

## Upload Helpers

### SerialBatchEventUploader
- Serialization: one POST in-flight maximum
- Batching: combines events respecting `maxBatchSize` and `maxBatchBytes`
- Backpressure: blocks when pending queue exceeds `maxQueueSize`

### WorkerStateUploader
Session state updates via `PUT /worker`:
- Patch coalescing merging updates while request in-flight
- RFC 7396 logic: last-value-wins for top-level keys; deep merge for metadata

## CCRClient (CCR v2)
High-level orchestrator managing SSETransport lifecycle:
- Stream accumulation coalesces `text_delta` chunks into "full-so-far" snapshots
- Idempotent snapshots reuse first chunk UUID for retry stability
- `StreamAccumulatorState` tracks session scopes and active message IDs

## Key Constants

| Constant | Value | Purpose |
|----------|-------|---------|
| `POST_TIMEOUT_MS` | 15,000ms | Single stuck POST blocking limit |
| `CLOSE_GRACE_MS` | 3,000ms | Write drainage period on close |
| `RECONNECT_GIVE_UP_MS` | 600,000ms | 10-minute reconnection budget |
| `LIVENESS_TIMEOUT_MS` | 45,000ms | SSE death detection threshold |
| `MAX_QUEUE_SIZE` | 100,000 | SerialBatchEventUploader memory bound |
