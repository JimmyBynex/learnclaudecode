# 3.1 Tool Interface & Permission System

## Overview
Standardized `Tool<Input, Output>` interface. Every capability (file reading, bash execution, etc.) is implemented as a Tool.

## Core Interface Elements

| Element | Purpose |
|---------|---------|
| `name` | Unique identifier for Anthropic API tool definition |
| `description` | Natural language explanation for model usage |
| `inputSchema` | JSON Schema defining expected arguments |
| `execute()` | Primary logic function accepting `ToolUseContext` |
| `validateInput()` | Optional custom validation beyond schema checks |
| `checkPermissions()` | Determines if execution requires user consent |

## Tool Factory
`buildTool()` simplifies tool creation with default behaviors for progress reporting and error handling.

## Permission System

### Three Permission Modes
1. **default**: Standard operation with prompts for sensitive actions
2. **bypass**: Internal mode allowing all actions without prompting
3. **yolo**: User-selected mode skipping all permission prompts

### Rule Evaluation (Three Sets)
- `alwaysAllowRules`: Matches patterns → automatic execution
- `alwaysDenyRules`: Blocks execution immediately on match
- `alwaysAskRules`: Forces user prompt regardless of safety classification

## ToolUseContext (passed to every `execute` call)

| Field | Purpose |
|-------|---------|
| `abortController` | Responds to user-initiated cancellations |
| `readFileState` | File state caching for optimized reads |
| `getAppState`/`setAppState` | Access to application state tracking |
| `setToolJSX` | Render custom React components in terminal |
| `options.mcpClients` | Access to MCP servers |

## Result Management
Large tool outputs truncated and stored in `ContentReplacementState`. Full content replaced with references to prevent context window overflow and API limit violations.
